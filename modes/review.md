# 审查模式（review）

维护现有知识库:补 capture/落库留下的粗糙、清遗留、盯库的健康趋势。**永不全量读库**——靠三件事:游标(只看变化的)、机械/判断分层、按切片。每轮产出一篇带日期的 report(格式见 `schemas/report.md`),report 兼作游标。

## 核心原则

- **review 永不一次性读全库**。只处理:游标之后变化的、或某个切片、或被 flag 的小集合。
- **机械 vs 判断分开**:能用脚本 blind 跑的(计数/grep/sed/index 同步/删打勾)绝不读全文;只有需要理解语义的(判重、评估该删该留、聚类升级)才读那几页。
- 低风险修复直接做;改语义、合并、删除前先问用户。
- **report 是最后一步**:所有事做完、log 写完,才落 report;report 存在 = 本轮完成。

## Step 1：读游标 + 定范围

- 读 `schemas/report.md`。**游标 = 最新一篇 report 的 `reviewed_at`**:`ls review/*/20*.md | sort | tail -1` 取其 frontmatter;`flags_open` 是本轮判断层的携带输入。库里还没有 report 时,问用户从哪个时间点开始。
- **"上次到哪、哪些要看" = log 就是变更流**:`grep '^## \[' log.md` 取游标之后的条目,它们提到的 `[[页]]` 就是"自上次以来动过的页" = 本轮判断层的默认输入。不用另维护清单。
- 按用户请求定切片(不指定就默认"游标之后变化的 + 跑一遍全库机械层"):只查某页 / 某类型 / 某簇(某创作者的一组页)/ scrap+dropzone / 游标之后变化的 / 全库健康(= 机械层扫全库,判断层仍只看 flag)。

## Step 2：机械层（脚本 blind,成本与库大小无关,可每次跑）

用 grep/sed/一次性脚本 + 计数/diff 验证,**不读全文**。`review/` 目录不参与任何机械检查:

1. **index.md 同步**:按 `wiki/` 各子文件夹扫每个文件 frontmatter 的 `title`+`tldr`,重新生成/校验 index 对应行——`title` 与文件名不同写 `[[文件名|title]]`、相同写 `[[文件名]]`(index 用裸名);summary 不进 index;`-全部作品` 台账不进;章节扁平不分组。新增补行、删除去行、`tldr` 改了同步。
2. **pending 清打勾 + 老化**:删所有 `- [x]`(含其下缩进子说明);统计未勾条目数和最老条目日期,进 report `metrics.pending_open` 和正文(挂太久的该提醒用户拍板或降级)。
3. **log 月度归档**:非当月条目按自然月合并成一条 `归档 · <YYYY-MM>`(不删除、只精简合并,保留页名/数量/topic);当月保持逐条。若同一 topic 几小时内被拆成多条,合并成一条。
4. **扫质量旗标**(grep,只报不改,进 `flags_open`):断链 `[[不存在的页]]`、孤儿页(无入链)、缺字段(无 `tldr`/来源/`updated`——排除 `-全部作品` 台账)、被反引号包住的 `[[]]`、正文里残留的全路径链接(正文该裸名)、`review_by` 已过期的 volatile 页。
5. **清遗留「编辑日志」时间线**:删 entity 页里 `## 时间线` / `## 版本里程碑` 这类**页内编辑日志**段——改动史由 git/log 承担。**保留真·内容时间线**(主体自己的历史:产品功能演进、话题发酵)。拿不准的先 flag、别删。
6. **raw 卫生**:`raw/inbox/` 积压计数(进 metrics);`raw/dropzone/` 老化(躺超过 30 天的条目数);wiki 页正文引用的 raw 路径是否存在(素材被移动/改名导致的死路径)。
7. **敏感信息扫描**(grep 模式,只报不改):API key 形态字符串、密钥、电话/证件号等误入 wiki 的个人敏感信息——知识库可能对外暴露(检索/API),这类内容不该在客观知识层。
8. **库指标快照**:各类型页数、index/log 行数、inbox 积压、pending 未决数——写进 report `metrics`,rollup 靠它出趋势。

## Step 3：判断层（agent 读,只读被 flag / 切片内的小集合，不读全库）

1. **重复/相似页**:标题、别名、同源。真重复才合并——合并=择优融合 + 互链 + 保留双方来源,不是简单删一个。
2. **冲突页**:同一事实两页说法不一 → 保留双方来源和日期、标冲突,不静默覆盖。
3. **scrap / dropzone 聚类升级**:反复出现的碎片能否升成正式 concept/method/entity。
4. **蒸馏质量抽检**:从游标后新建/更新的页里抽少量样本,对照蒸馏手艺标准(是"拆知识点"还是"总结原文"?能否脱离原文独立存活?绝对日期?来源指针?),问题进 report,系统性偏差建议改 distill 手艺章节。
5. **互链与承接页缺口**:同一工具/概念在多页反复出现却互不链接、或多个 entity 都在提某个还没有页的概念(该建承接页了)——提建议,批量新建先问用户。
6. **过期强实时页**:`valid_as_of` 老、`volatility: volatile` 的,标状态或提示回查。
7. 逐个补机械层报的旗标(断链/孤儿/缺字段);处理不完的留 `flags_open` 滚动携带。

## Step 4：收尾（顺序固定）

1. 低风险修复与结构操作写一条 **一行版** review log(指向即将写的 report:`详见 [[review/YYYY-MM/...]]`)。
2. 需用户确认的合并/删除结论 → `pending.md`。
3. git 提交本轮修复。
4. **最后写 report**(`schemas/report.md`):frontmatter 数字如实统计,`reviewed_at` 用本机本地时间,`flags_open` 写剩余旗标;正文五节(发现/已处理/需要决定/趋势/下一步)。report 落盘后本轮才算完成。
5. 若发现该新增类型/规则,说明理由并建议改对应 `references/`/`schemas/`。

## 输出

把 report 的要点复述给用户:发现(按严重程度)、已处理、需要决定、趋势异常、下一步建议。
